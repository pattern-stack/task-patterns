# Task Management Templates & Guidelines

## Overview

This document provides templates and guidelines for creating Epics, Tasks, and Subtasks in Linear, following established patterns for AI-human collaborative development.

## Ticket ID System

- **Format**: `[TEAM]-[NUMBER]` (e.g., `TASK-19`, `BE-45`, `ETL-123`)
- **Teams**: TASK, BE, ETL, TECH, DEV, etc.
- **Numbers**: Auto-generated by Linear

## Quick Reference

### When to Create an Epic
- Multiple related features (5+ tasks)
- Cross-team coordination required
- Multi-week implementation
- Strategic initiative
- Major refactoring effort

### When to Create a Task
- Specific deliverable
- 1-5 days of work
- Clear acceptance criteria
- Can be assigned to individual
- Testable outcome

### When to Create a Subtask
- Part of larger task
- 1-4 hours of work
- Atomic unit that can be completed independently
- Helps track progress on complex tasks

## Workflow States

1. **Backlog** - Not yet prioritized
2. **Todo** - Ready to start
3. **In Progress** - Actively working
4. **In Review** - Code review/QA
5. **Blocked** - Waiting on dependency
6. **Completed** - Done and verified
7. **Canceled** - No longer needed

## Best Practices

### Writing Good Titles
- **Epic**: High-level capability (e.g., "Implement Authentication System")
- **Task**: Specific deliverable (e.g., "Add JWT token validation to API endpoints")
- **Subtask**: Atomic action (e.g., "Write unit tests for auth middleware")

### Estimation Guidelines
- **XS (1pt)**: < 2 hours
- **S (2pt)**: 2-4 hours
- **M (3pt)**: 4-8 hours (1 day)
- **L (5pt)**: 2-3 days
- **XL (8pt)**: 3-5 days
- **XXL (13pt+)**: 1+ week (consider breaking down)

### Using tp CLI
```bash
# Create tasks during development
tp add "Implement user authentication" --team BE

# Track progress
tp working BE-45
tp done BE-45

# View context
tp context

# Update task
tp update BE-45 --status "In Review"
```

### TDD Workflow Integration
1. Create task: `tp add "Add validation to user service"`
2. Mark in progress: `tp working TASK-XX`
3. Write failing tests first (RED)
4. Implement code to pass tests (GREEN)
5. Refactor and optimize (REFACTOR)
6. Mark complete: `tp done TASK-XX`

## Tips for AI Collaboration

### When Working with Claude
1. Reference tickets by ID (e.g., "Let's work on TASK-19")
2. Use `tp context` to sync on current state
3. Create tasks during conversation with `tp add`
4. Include ticket ID in commit messages
5. Use Co-Authored-By in commits

### Context Sharing
```bash
# Share current context with AI
tp ai-context

# Show specific task details
tp show TASK-19
```

## Common Patterns

### Feature Development Epic
1. Research & Design (Task)
2. Backend Implementation (Task)
   - Database schema (Subtask)
   - API endpoints (Subtask)
   - Service logic (Subtask)
3. Frontend Implementation (Task)
4. Testing & QA (Task)
5. Documentation (Task)
6. Deployment (Task)

### Bug Fix Task
1. Reproduce issue (Subtask)
2. Write failing test (Subtask)
3. Implement fix (Subtask)
4. Verify fix (Subtask)
5. Update documentation (Subtask)

### Refactoring Epic
1. Identify code smells (Task)
2. Design new architecture (Task)
3. Implement changes (Task)
   - Module 1 refactor (Subtask)
   - Module 2 refactor (Subtask)
4. Update tests (Task)
5. Performance validation (Task)
